# MR Description Generator

Скилл создаёт описание Merge Request на русском языке и публикует его через `glab`.

---

## Шаг 1 — Разбери входные данные

Пользователь вызывает скилл с URL вида:
```
/mr-desc https://selfhostedgitlab/namespace/repo/-/merge_requests/123
```

Из URL извлеки:
- **host** — `selfhostedgitlab`
- **namespace/repo** — `namespace/repo`
- **mr_id** — `123`

Сохрани эти значения — они понадобятся при отправке через `glab`.

---

## Шаг 2 — Получи данные MR из GitLab

Используй `glab` для получения метаданных MR (ветка, цель, заголовок):

```bash
glab mr view <mr_id> --repo <host>/<namespace>/<repo>
```

Это даст тебе:
- исходную ветку MR (`source_branch`)
- целевую ветку (`target_branch`, обычно `main` или `master`)
- текущий заголовок MR

---

## Шаг 3 — Собери git-контекст

Определи репозиторий локально (текущий `git remote`) и собери diff между ветками:

```bash
# Список коммитов в ветке MR
git log origin/<target_branch>..origin/<source_branch> --oneline --no-merges

# Изменённые файлы с типом изменения
git diff --name-status origin/<target_branch>...origin/<source_branch>

# Статистика изменений
git diff --stat origin/<target_branch>...origin/<source_branch>

# Полный diff
git diff origin/<target_branch>...origin/<source_branch>
```

Если локального репозитория нет или ветки недоступны — используй только данные из `glab mr view`.

Если diff большой (>500 строк) — читай точечно по ключевым файлам:
```bash
git diff origin/<target_branch>...origin/<source_branch> -- <path/to/file>
```

---

## Шаг 4 — Проанализируй изменения

Перед написанием ответь себе на вопросы:

1. **Что** изменилось? (новый функционал, баг, рефакторинг, конфиг, документация, тесты)
2. **Почему** это было нужно? (контекст, проблема)
3. **Как** реализовано? (ключевые решения, паттерны, библиотеки)
4. **Что может сломаться?** (breaking changes, миграции, переменные окружения)
5. **Как проверить?** (шаги для ревьюера)

Определи **тип изменений**: `feat` | `fix` | `refactor` | `chore` | `docs` | `test` | `perf`

---

## Шаг 5 — Сформируй описание

Генерируй текст строго по шаблону. Не добавляй пустые секции — если секция неприменима, просто её опусти.

```markdown
## 📋 Описание

<2–4 предложения: ЧТО сделано и ЗАЧЕМ. Контекст, а не пересказ кода.>

## 🔧 Тип изменений

- [ ] ✨ Новая функциональность (feat)
- [ ] 🐛 Исправление бага (fix)
- [ ] ♻️ Рефакторинг (refactor)
- [ ] 📦 Зависимости / конфигурация (chore)
- [ ] 📝 Документация (docs)
- [ ] ✅ Тесты (test)
- [ ] ⚡ Производительность (perf)

## 📝 Что изменено

- **<Модуль/компонент>**: <что сделано и зачем>
- **<Модуль/компонент>**: <что сделано и зачем>

## 💡 Технические детали

<Только нетривиальные решения: архитектурный выбор, обход ограничения, специфичный API.
Если изменения очевидны — секцию опустить.>

## ⚠️ Breaking Changes

<Изменения, ломающие обратную совместимость: API, схема БД, env-переменные, конфиги.
Если нет — секцию опустить.>

## 🗄️ Миграции / Конфигурация

<Миграции БД, новые переменные окружения, действия при деплое.
Если нет — секцию опустить.>

## ✅ Как проверить

1. <шаг>
2. <шаг>
3. Ожидаемый результат: <что должно произойти>

## 🧪 Тесты

<Какие тесты покрывают изменения, что добавлено/изменено.
Если тестов нет — укажи причину. Если неприменимо — опусти.>

## 🔗 Связанные задачи

<Ссылки на тикеты/issues. Если нет — секцию опустить.>
```

---

## Шаг 6 — Интерактивный цикл

После того как описание готово, выведи его целиком в markdown-блоке, затем **всегда** задай вопрос с тремя вариантами:

---

**Что делаем дальше?**

1. 🚀 **Отправить** — опубликовать описание в MR `!<mr_id>`
2. ✅ **Готово** — описание финальное, не отправлять
3. ✏️ **Улучшить** — введи свой запрос на доработку

---

### Если пользователь выбрал «Отправить» (1)

Запиши описание во временный файл и отправь через glab:

```bash
cat > /tmp/mr_description.md << 'EOF'
<полный текст описания>
EOF

glab mr update <mr_id> \
  --repo <host>/<namespace>/<repo> \
  --description "$(cat /tmp/mr_description.md)"
```

После успешной отправки сообщи: «✅ Описание опубликовано в MR !<mr_id>».

### Если пользователь выбрал «Улучшить» (3)

Прими свободный текст (например: «сделай тон формальнее», «добавь про кэширование», «сократи»).
Доработай описание и снова выведи результат + вопрос из Шага 6.
Повторяй цикл до выбора «Отправить» или «Готово».

---

## Правила написания

**Делай:**
- Пиши на русском, технические термины (API, endpoint, middleware) оставляй как есть
- Объясняй «зачем», а не только «что» — контекст ценнее перечисления файлов
- Группируй изменения по логическому модулю, не по именам файлов
- Отмечай нужные чекбоксы: `[x]`

**Не делай:**
- Не пиши «изменён файл `src/utils.ts`» — это видно в диффе
- Не копируй сообщения коммитов без обработки
- Не оставляй заглушки типа `<TODO>` или пустые секции
- Не пиши «minor fixes» без конкретики
